# Repository Overview

This repository contains a Model Context Protocol (MCP) server that exposes the Polarion REST API to AI assistants. It is generated from Polarion's OpenAPI specification and targets Polarion 2506 only. See [polarion-versioning.md](./polarion-versioning.md) for version support.

## What This Server Does

- Provides an MCP stdio server for AI assistants.
- Provides an optional HTTP server for Custom GPTs and other HTTP clients.
- Maps MCP tools to Polarion REST API operations.
- Adds Polarion-specific resources, prompts, and project-configuration caching around the generated tool set.

## Repository Layout

Root holds only entry points and tooling config; everything else is in folders.

- src/ - TypeScript source code (entry points listed below).
- tests/ - Native Node.js test suite for handwritten logic.
- docs/ - Repository documentation (incl. example-task.md, a worked end-to-end example).
- scripts/ - Maintainer helpers: generate-tools.mjs (extracts src/tools.ts from the generator output), fix-sparse-fields.cjs (OpenAPI patcher), generate-api-key.sh (HTTP API key generator).
- sdk/ - Placeholder for optional local Polarion PDFs (not bundled; see sdk/README.md).
- .github/ - CI workflow (ci.yml) and CONTRIBUTING.md.
- build/ - Compiled output (generated by `npm run build`; not committed).
- Root config: package.json / package-lock.json (npm), tsconfig.json / tsconfig.test.json (TypeScript), Dockerfile, render.yaml (Render blueprint, must stay at root), .env.example, and the dotfiles (.dockerignore/.gitignore/.gitattributes).

## Entry Points

- src/server.ts - Shared MCP server factory (tools/resources/prompts); used by every transport.
- src/index.ts - stdio MCP entry point (local clients).
- src/mcp-http-server.ts - Streamable HTTP MCP transport (remote clients, e.g. Claude.ai).
- src/http-server.ts - REST HTTP wrapper for ChatGPT Custom GPT Actions.
- src/index-http.ts - REST HTTP entry for deployment platforms.

## Handwritten vs Generated Code

- Handwritten modules implement transports, validation, auth, documentation resources, prompts, and deployment behavior.
- src/tools.ts is generated from Polarion OpenAPI and should be refreshed through the documented generation workflow instead of edited manually.

## Generated Assets

- src/tools.ts is generated (toolDefinitionMap + securitySchemes) and should not be edited manually; regenerate with `npm run regenerate`.
- openapi.json and .gen/ are gitignored regeneration intermediates (default spec: public Polarion demo, override via SPEC_URL).
- build/ is created during compilation and is not committed.

## Distribution

- The CI workflow builds a slim, runnable release ZIP (`polarion-mcp-v<version>.zip`) on every push to main and uploads it as a build artifact.
- The ZIP contains build/, package.json, package-lock.json, sdk/, README.md, LICENSE, and .env.example.
- To run it: unzip, `npm ci --omit=dev`, set environment variables, then `node build/index.js` (stdio) or `node build/mcp-http-server.js` (Streamable HTTP / Claude.ai).
- The same workflow verifies the ZIP by unzipping it, installing production dependencies, starting the server, and running an MCP initialize round-trip.

## Validation Surface

- npm run build compiles the runtime output.
- npm run typecheck verifies TypeScript without emitting.
- npm test runs the unit tests for handwritten logic.
- The CI `package` job additionally builds and smoke-tests the release ZIP.
